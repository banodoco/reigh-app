When a user resizes a video clip by dragging its edge, the system currently adjusts `from`/`to` (source trim points) without changing `speed`. This causes freeze-frame when extended beyond source duration. The fix: video clips adjust speed (preserving full source playback), audio clips clamp to source duration and never change speed. Key formula: `newSpeed = sourceDuration / newTimelineDuration` where `sourceDuration = originalTo - originalFrom`.

        - [CRITICAL] Stale clipMeta.to still not addressed — plan reads it in onActionResizeStart for speed formula
  The v1 review flagged that clipMeta.to is INCORRECT after a left-edge resize (current code at line 67 computes to using the OLD from, producing a wrong value that rowsToConfig silently recomputes away). The v2 plan Step 1 says to store `to` in resizeStartRef but never specifies how to compute it — the natural reading is `clipMeta.to`, which is stale. Example: clip start=5,end=15,from=0,to=10,speed=1 after left resize to start=7 has clipMeta.to=8 (wrong, correct is 10). Speed formula `(origin.to - origin.from) / duration` would use 8-2=6 instead of 10-2=8. FIX: In onActionResizeStart, compute to via `getSourceTime({ from: clipMeta.from ?? 0, start: action.start, speed: clipMeta.speed ?? 1 }, action.end)` — not from clipMeta.to.
  Source: useClipResize.ts:65-73 (stale to computation), plan Step 1.2
